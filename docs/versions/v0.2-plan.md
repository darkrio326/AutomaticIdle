# v0.2 版本计划

更新日期：2026-03-30
文档状态：Execution Plan

---

## 1. 版本定位

- **版本号**：v0.2
- **承接版本**：v0.1（runtime 实时引擎 + UI 三列布局已落地）
- **版本主题**：内容扩展入口 — 建筑 · 工具 · 动态订单
- **核心目标**：
  1. **建筑解锁系统**：玩家通过消耗资源建造建筑，解锁新的资源链与配方（铜矿→铜锭、煤矿→钢），形成明确的进度节点控制；
  2. **工具链系统**：玩家通过消耗资源购买工具，对指定配方施加 timeMultiplier 效率加速，产生"资源→投资→加速→更高产出"正反馈循环；
  3. **动态订单系统**：将当前静态订单列表改造为"随机生成 + 槽位管理 + 时限 + 刷新 + 奖励分层"的动态系统，引入每轮外部驱动变量；
  4. **数据层扩展**：将 buildings.json / tools.json / sells.json 正式接入 GameConfig 类型与引擎加载逻辑。

## 4. 阶段拆分

| 阶段 | 目标 | 输出物 | 风险 |
|------|------|--------|------|
| Phase 1 | 数据层扩展：将 buildings / tools / sells 正式接入类型系统与加载逻辑 | `BuildingConfig` / `ToolConfig` 新增接口；`GameConfig` 增加三字段；`flowStore` 合并 sells 到配方可选列表 | 合并 sells 后配方 ID 冲突（低风险，已规范命名）|
| Phase 2 | 建筑解锁系统：购买建筑解锁新资源/配方，形成进度控制器 | `buildingStore.ts`（购买状态、解锁状态）；建筑 UI 面板；引擎接入 enabled 配方/资源同步 | 解锁节奏不合理导致卡死或一步到位无目标 |
| Phase 3 | 工具链系统：购买工具对指定配方施加时间加速 | `toolStore.ts`（装备状态、效果应用）；工具 UI 面板；引擎 `calcActualStepTime` 接入工具效果叠加 | 多工具倍率叠加后数值爆炸 |
| Phase 4 | 动态订单系统：随机生成 + 有限槽位 + 时限 + 刷新 + 分层奖励 | `orderStore.ts` 重构（生成器 + 槽位管理 + 计时器）；订单 UI 更新（槽位展示/倒计时/删除） | 随机性过强导致不公平；时限过严造成焦虑感 |
| Phase 5 | P2 补全：流程模板 + Runtime 调试面板（可选，视 Phase 1~4 完成情况决定） | `flowTemplateStore.ts`（模板保存/读取/切换）；调试面板（手动推进步骤、状态注入） | Phase 5 对核心玩法无直接影响，可延至 v0.3 |

## 7. 验收与回滚

### 验收标准

1. **建筑系统**：从零资源出发，通过采矿→出售→购买铁匠铺，能解锁铁剑锻造；购买浅层矿井后可开采铜矿；购买高温炉后可熔炼钢。
2. **工具系统**：购买铜镐后 mine_iron / mine_copper 耗时明显缩短，GPS 可观察到提升；购买多件工具后效率叠加但不超出合理范围。
3. **动态订单**：启动后订单区展示 3 张卡片，倒计时正常流逝，到期消失，延迟后补充；提交订单后奖励正确发放；资源出售/建筑/工具是合理的奖励类型（不全是金币）。
4. **数据一致性**：sells.json 中的出售配方可在流程编辑器"出售"分组中选到并正常执行；建筑/工具购买后存档恢复时状态完整保留。
5. **类型检查通过**：`vue-tsc --noEmit` 无报错。
6. **构建通过**：`npm run build` 成功。

### 回归范围

| 模块 | 验证点 |
|------|--------|
| 流程编辑 | sells.json 配方可添加为步骤并正常执行（timeSeconds: 0 步骤不阻塞） |
| 建筑购买 | 购买解锁后对应配方/资源立即可用；刷新后状态保留 |
| 工具购买 | 购买后 calcActualStepTime 反映工具加速；不与 upgrades 系统冲突 |
| 订单系统 | 旧的静态订单行为不退化；动态生成正常；到期/删除/刷新逻辑正确 |
| 存档恢复 | 建筑/工具购买状态正确持久化；动态订单槽位状态（含倒计时剩余）正确恢复 |
| 离线收益 | 引入工具/建筑后离线收益计算仍正确 |

### 回滚策略

- 每个 ITER 单独提交；Phase 1 改动全为扩展，不删除现有接口
- ITER-022 向前兼容：现有 GameConfig 使用方不受影响（新字段使用 `buildings?` / `tools?` 可选字段过渡）
- ITER-027 动态订单重构：在合并前保留旧版 `OrderPanel.vue`，验收后再删除旧版

## 8. 数值平衡约束

> 本节作为设计约束，供逐 ITER 实施时参考，不作为验收硬标准。

| 系统 | 约束原则 |
|------|---------|
| 建筑费用 | 每件建筑费用应在玩家顺畅运行 3~10 分钟后可负担（基于 iron_ingot 产出速率估算） |
| 工具 timeMultiplier | 单件工具不低于 0.5x（最多缩短 50%）；所有工具叠加不低于 0.2x |
| 订单时限 | 初版订单存续时间 >= 10 分钟，偏宽松，避免造成焦虑 |
| 订单奖励 | 高价值订单（需要稀有材料）奖励必须包含解锁内容或效率提升，不应仅为金币 |
| 资源用途分化 | copper_ingot 主要用于工具投资（而非直接出售），钢主要用于高级工具和建筑 |
