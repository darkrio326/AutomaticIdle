- 描述：将当前原型从“静态单轮计算结果展示”升级为“持续运行中的系统表现”。界面需要体现流程正在自动循环执行，当前步骤有明确高亮和进度，资源数值持续变化，金币/材料/EXP 以实时增长的方式反馈给玩家。流程修改后应即时生效，建议第一版采用“当前步骤执行完成后，于下一轮生效”的规则，以降低实现复杂度并保持行为稳定。技术上应从一次性 simulateFlow 扩展为 tick 驱动的 runtime loop：时间流逝 → 当前步骤进度推进 → 完成动作 → 产出资源与经验 → 切换到下一步骤 → 循环执行。UI 层需要重点表现：当前执行步骤高亮、步骤进度条、资源数字连续跳动、每秒收益实时变化、瓶颈提示实时刷新。
- 价值：这是把产品从“流程收益计算器”拉到“真正有运行感的放置游戏”的关键一步。玩家会明显感觉自己搭建的是一个正在运转的自动化系统，而不是静态表格。该能力会显著提升沉浸感、反馈爽感和流程优化意愿。
- 风险：如果直接采用“修改流程立即打断当前步骤并重算”的方案，会显著增加状态同步与边界处理复杂度；如果资源数值只做低频刷新而非连续反馈，运行感会不足；如果 runtime loop 与 store/UI 强耦合，后期扩展会变得困难。
- 结论：纳入第一阶段交互增强范围，优先于复杂视觉包装。
- 原因：该能力直接决定产品是“工具型原型”还是“有游戏感的系统”。在核心模拟器已经完成的前提下，下一步最值得投入的是 runtime loop 与动态反馈，而不是先做复杂美术或流程图拖拽。
